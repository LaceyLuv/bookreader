# BookReader 속도 개선 및 용량 축소 기획안

## 1. 목적

`C:\dev\bookreader`의 현재 구조를 기준으로, `C:\Users\Lenovo\Downloads\texview190` 대비 실행 속도 개선과 설치 용량 축소를 위한 실행계획을 정의한다.

본 문서는 코드 구현안이 아니라 우선순위, 계측 기준선, 구조 개선 방향, 수동 검증 기준을 정리한 계획 문서다.

## 2. 기준선 요약

로컬 확인 기준:

- `texview190` 전체 크기: 약 `2.64 MB`
- BookReader 프런트 정적 산출물 `frontend/dist`: 약 `0.41 MB`
- BookReader Python sidecar EXE: 약 `36.24 MB`
- BookReader NSIS 설치 파일: 약 `38.59 MB`

핵심 해석:

- 현재 용량 문제의 본체는 프런트엔드가 아니라 `PyInstaller + Python 런타임 + FastAPI/Uvicorn/EPUB 파서`를 통째로 묶은 sidecar다.
- 설치 파일 크기는 사실상 sidecar EXE가 결정하고 있다.
- `texview190`과 동일 수준의 절대 용량을 그대로 맞추는 것은 현 아키텍처상 비현실적이다. 대신 1차 목표는 `설치 파일 20 MB대`, 2차 목표는 `15 MB대 진입 여부 검토`로 잡는 것이 현실적이다.

## 3. 현 구조에서 보이는 병목

### 3.1 패키지/설치 용량 병목

- `backend/bookreader-backend.spec`에서 `uvicorn`의 HTTP/WebSocket/lifespan 경로를 넓게 포함한다.
- Python sidecar가 `ebooklib`, `bs4`, `lxml`과 런타임 전체를 포함한다.
- `books`, `fonts` 폴더를 sidecar 데이터로 묶고 있어 배포 산출물 비대화 가능성이 있다.
- 데스크톱 앱은 Tauri shell + 외부 Python sidecar 2중 런타임 구조다.

### 3.2 시작 속도 병목

- 데스크톱 실행 시 Tauri가 sidecar를 띄우고, sidecar가 다시 FastAPI/Uvicorn 서버를 부팅한다.
- 서버 기반 구조라 앱 UI 준비와 API 준비가 분리되어 초기 체감 지연이 생길 가능성이 높다.
- `main.py`의 lifespan에서 라이브러리/어노테이션 저장소 초기화가 시작 시점에 수행된다.

### 3.3 TXT 리더 병목

- `backend/services/txt_service.py`는 TXT 파일 전체를 메모리로 읽고 디코드한다.
- `frontend/src/components/TxtReader.jsx`는 전체 텍스트를 한 번에 DOM에 넣고 컬럼 기반 페이지 수를 계산한다.
- 긴 TXT일수록 초기 로드, 재측정, 검색 하이라이트 비용이 커진다.

### 3.4 EPUB 리더 병목

- `backend/services/epub_service.py`는 EPUB 파싱 결과를 캐시하지만 챕터 HTML 가공과 스타일 재작성 비용이 남아 있다.
- `frontend/src/components/EpubReader.jsx`는 숨겨진 측정 DOM을 별도로 만들고, 이미지와 폰트 로딩 완료를 기다린 뒤 페이지 수를 계산한다.
- 챕터 수가 많은 EPUB에서 전체 페이지 수를 확보하기 위한 백그라운드 측정 비용이 크다.

### 3.5 ZIP 리더 병목

- `backend/services/zip_service.py`는 이미지 요청마다 ZIP을 열고 파일을 읽는다.
- `frontend/src/components/ZipReader.jsx`는 현재/다음 이미지만 단순 로드하며, 압축 해제 캐시·프리패치·썸네일 전략이 없다.

## 4. 성능/용량 목표

### 4.1 1차 목표: 2주 내 빠른 개선

- 설치 파일: `38.6 MB -> 24~28 MB`
- 앱 시작 체감: 메인 창 표시 후 읽기 가능한 상태까지 `30% 이상 단축`
- TXT 대용량 파일 첫 페이지 표시: `25% 이상 단축`
- EPUB 첫 챕터 표시: `20% 이상 단축`

### 4.2 2차 목표: 구조 개선 완료 후

- 설치 파일: `15~20 MB` 범위 검토
- sidecar 단독 EXE: `36.2 MB -> 12~18 MB`
- ZIP 이미지 넘김 시 다음 페이지 대기 감소
- EPUB 전체 페이지 수 계산을 즉시 완료형이 아니라 점진 완료형으로 전환

## 5. 우선순위별 실행계획

## Phase 0. 계측 기준선 확정

목적:

- 감으로 최적화하지 않고 릴리즈마다 비교 가능한 수치를 만든다.

산출물:

- 기준 시나리오 3종: 대용량 TXT, 이미지 많은 EPUB, ZIP 만화
- 측정 지표 5종:
  - 앱 시작 시간
  - 책 열기 완료 시간
  - 페이지 전환 시간
  - 메모리 피크
  - 설치 파일/sidecar 크기

권장 측정 방식:

- 웹:
  - 브라우저 Performance 패널
  - 네트워크 응답 시간
  - 첫 콘텐츠 표시 시점
- 데스크톱:
  - 프로세스 시작 후 `/api/health` 응답 가능 시점
  - 책 열기 후 첫 렌더 시점
  - sidecar 메모리 사용량

완료 기준:

- 동일 샘플 파일과 동일 장비에서 릴리즈 전후 수치 비교표 확보

## Phase 1. 용량 축소 우선

목적:

- 구현 난도가 낮고 효과가 큰 패키징 비대화부터 줄인다.

과제:

1. PyInstaller 포함 모듈 축소
- `uvicorn[standard]` 대신 필요한 최소 런타임만 남기는 방향 검토
- WebSocket, 개발 편의성 모듈, 불필요한 transport 경로 제외
- `backend/bookreader-backend.spec`의 `hiddenimports` 최소화

2. sidecar 번들 방식 재검토
- `onefile` 유지가 실제로 압축에 유리한지 재검증
- `onedir + NSIS solid compression`과 크기/시작속도 비교
- `console=True`가 필요 없다면 창 없는 실행으로 전환 검토

3. sidecar 데이터 분리
- 배포본에 기본 `books` 데이터가 꼭 필요하지 않다면 제외
- `fonts`는 기본 포함 대신 사용자 추가형으로 전환 검토

기대효과:

- 가장 적은 코드 수정으로 설치 파일과 sidecar 크기 직접 감소

## Phase 2. 초기 체감속도 개선

목적:

- 사용자가 느끼는 "실행 후 바로 읽을 수 있음"을 개선한다.

과제:

1. 시작 시 동기 초기화 축소
- 라이브러리/어노테이션 저장소 초기화를 lazy load 또는 백그라운드화
- 첫 화면에 필요한 API만 우선 준비

2. 첫 화면 우선 렌더
- 대시보드에서 전체 라이브러리 메타데이터를 한 번에 읽지 않도록 제한
- 최근 책/빠른 진입만 먼저 렌더하고 나머지는 점진 로딩

3. sidecar 기동 체계 단순화 검토
- FastAPI 서버 프로세스 자체를 유지할지 검토
- 데스크톱 전용 경로는 Tauri command 또는 Rust 얇은 브리지로 대체 가능한 범위 정의

기대효과:

- 설치 용량보다 먼저 체감되는 불만인 "실행이 굼뜸" 해소

## Phase 3. 포맷별 리더 최적화

목적:

- TXT/EPUB/ZIP 각각의 병목을 별도 해소한다.

### TXT

- 전체 파일 로드 대신 청크 단위 분할 캐시 검토
- 검색용 인덱스와 렌더용 텍스트를 분리
- 전체 DOM 주입 대신 가시 범위 중심 pagination 또는 virtual paging 검토

예상효과:

- 대용량 TXT 첫 페이지 표시 속도와 메모리 사용량 개선

### EPUB

- 챕터 HTML 가공 결과 캐시 범위 확대
- 전체 페이지 수 계산을 백그라운드 점진 집계형으로 고정
- 이미지 로딩 대기 없는 임시 페이지 수와 확정 페이지 수 2단계 모델 검토
- embedded font 사용 시점 최소화

예상효과:

- 첫 챕터 로드 지연 감소
- 챕터 이동 시 재측정 비용 감소

### ZIP

- ZIP 핸들 재사용 또는 이미지 바이트 캐시 도입 검토
- 현재 페이지 기준 앞뒤 1~2장 프리패치
- 큰 이미지에 대해 축소 캐시 또는 디코딩 힌트 적용 검토

예상효과:

- 만화 넘김 시 끊김 감소

## 6. 구조 개선 방향

### Option A. 현 구조 유지 + 최소 최적화

내용:

- Python sidecar 유지
- PyInstaller/초기화/리더 로직만 다이어트

장점:

- 기존 API와 웹/데스크톱 공용 구조 유지

단점:

- 용량 절감 한계가 뚜렷함

권고:

### Option B. 데스크톱 경량화 전용 백엔드 재구성

내용:

- 웹은 FastAPI 유지
- 데스크톱은 Tauri command + Rust/로컬 라이브러리 호출로 일부 API 대체

장점:

- 설치 파일과 시작속도 모두 크게 개선 가능

단점:

- 구현 난도 상승
- 웹/데스크톱 경로 이원화 관리 필요

권고:

- Phase 1~2 수치 확인 후 투자 여부 결정

### Option C. Python sidecar 제거

내용:

- TXT/ZIP/EPUB 처리를 Rust 또는 JS 기반으로 재작성

장점:

- 장기적으로 가장 작은 설치 파일 가능

단점:

- 재구축 비용이 크고 현재 MVP 속도와 충돌

권고:

- 현재 라운드 범위 밖. 장기 과제로만 유지

## 8. 의사결정 기준

아래 조건을 충족하면 다음 단계로 간다.

- Phase 1만으로 설치 파일이 `30 MB 이하`로 내려가면 Python sidecar 유지 전략 지속
- Phase 1 후에도 `30 MB 이상`이고 시작속도 개선폭이 작으면 데스크톱 전용 백엔드 경량화 설계 착수
- TXT/EPUB 대용량 파일에서 첫 페이지 표시가 여전히 느리면 포맷별 가상화 설계를 별도 스프린트로 분리

## 9. 수동 검증 계획

### 웹 경로

1. `backend/run_server.py`로 API 실행
2. `frontend`에서 `npm run dev` 실행
3. TXT 대용량 파일 업로드 후 첫 화면 표시 시간 기록
4. EPUB 업로드 후 TOC 표시, 첫 챕터 표시, 챕터 전환 시간 기록
5. ZIP 업로드 후 첫 이미지 표시와 다음 장 넘김 지연 기록
6. 브라우저 메모리와 long task 발생 여부 기록

### 데스크톱 경로

1. `frontend`에서 `npm run desktop:build` 기준으로 설치본 생성
2. 설치 파일 크기, sidecar EXE 크기 기록
3. 앱 실행 후 메인 창 표시 시점 기록
4. `/api/health` 응답 가능 시점 기록
5. TXT/EPUB/ZIP 각각 첫 표시 시간과 페이지 전환 지연 기록
6. 앱 종료 후 sidecar 프로세스 종료 여부 확인

## 10. 결론

현재 BookReader의 용량 문제는 프런트엔드가 아니라 Python sidecar 중심 구조에서 발생한다. 따라서 이번 기획의 핵심은 다음 두 가지다.

- 단기: PyInstaller 번들 다이어트와 초기화 지연 제거
- 중기: 데스크톱 전용 경량 실행경로를 별도 설계할지 수치로 판단

가장 현실적인 첫 목표는 `38.6 MB -> 20 MB대` 진입과, TXT/EPUB 첫 표시 체감시간 `20~30%` 개선이다.
